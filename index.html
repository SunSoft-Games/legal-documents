Filosofía del estudio

Un estudio de una sola persona, apalancado por un enjambre de agentes de IA, operando bajo estándares estrictos de arquitectura de software y con foco quirúrgico en Time-to-Market.

Misión

Establecer un estudio de videojuegos móviles hiper-casuales de alta rentabilidad, optimizando costos y tiempos mediante desarrollo agéntico basado en IA — más de 10 años de arquitectura fullstack humana dirigiendo un enjambre de agentes que programan, auditan y documentan.

Público objetivo: casual gamers globales — mecánicas simples, loops de enganche cortos, fáciles de aprender, difíciles de dominar. Minimizar Time-to-Market sobre desarrollo complejo a largo plazo.

Modelo operativo

🧭 El Arquitecto (Humano)

  • Dirección de producto y decisiones comerciales finales
  • Revisión de seguridad y optimización de rendimiento
  • Publicación en tiendas y configuración de plataformas

🤖 El Enjambre de Agentes (IA)

  • Ideación de juegos y análisis de mecánicas competitivas
  • Generación de lógica de juego, CSS y resolución de bugs
  • Assets estéticos, sprites placeholder y SFX sintetizados

🎯 Metas Corto Plazo

Boilerplate estándar + 3 MVPs publicados en Google Play para validar retención, gasto < $300 USD.

🚀 Metas Largo Plazo

02 · Tecnología

Stack Tecnológico Restringido

Enfoque "Vanilla Fast": cero dependencias pesadas, portabilidad total, rendimiento predecible en cualquier gama de dispositivo.

Core tecnológico

HTML5 Canvas 2D Vanilla JS ES6+ CSS3 Flexbox/Grid Web Audio API Apache Cordova LocalStorage

Prohibido: Unity, Three.js, PixiJS, React/Angular/Vue dentro del motor de juego, Webpack/Vite/Rollup en runtime. Todo el audio se sintetiza en tiempo real — cero archivos de sonido remotos.

Game Loop estándar

// Nunca setInterval() — siempre rAF + Delta Time
let lastTime = 0;

function gameLoop(timestamp) {
  const dt = timestamp - lastTime;
  lastTime = timestamp;

  update(dt);
  render();

  requestAnimationFrame(gameLoop);
}

Contrato de arquitectura por juego

Juego-X/
├── index.html
├── css/
│   └── styles.css
└── js/
    ├── main.js    // loop + eventos
    ├── Game.js    // estado global
    ├── Engine.js  // render/colisión
    ├── Assets.js  // audio + assets
    └── Entities/
  • Object Pooling obligatorio para balas, partículas, explosiones y entidades temporales — cero new dentro de update()/render().
  • Resolución lógica fija (1080×1920 vertical) — nunca se reescala para el dispositivo, solo el canvas contenedor.
  • Persistencia 100% LocalStorage — sin backend/nube durante el MVP.
  • Renderer puro: render() nunca muta estado, solo dibuja.

CLAUDE.md · Rulebook

Arquitectura y Estándares de Ingeniería

Más de 115 reglas ARQ numeradas gobiernan cada decisión técnica del catálogo — desde el bucle de renderizado hasta la validación de recibos de compra. Estas son las columnas vertebrales.

Núcleo del motor (ARQ-001 → 033)

ARQ-010

Flujo Input → Update(dt) → Render

Tres etapas secuenciales estrictas por frame; el audio se dispara dentro de UPDATE sin bloquear el render.

ARQ-011

Renderer puro

Prohibido mutar estado, instanciar objetos o calcular colisiones dentro de render().

ARQ-031

Object Pooling obligatorio

Partículas, fragmentos y VFX masivos usan pools pre-asignados de tamaño fijo — cero presión sobre el GC móvil.

ARQ-032

Soporte DPR

devicePixelRatio siempre calculado para nitidez absoluta en pantallas OLED/Retina sin perder rendimiento.

Identidad visual y economía (ARQ-050 → 117)

ARQ-054

Glassmorphism OLED

Superficies casi transparentes + backdrop-filter: blur() sobre fondo negro OLED puro — el lenguaje visual de todo el catálogo.

ARQ-105

Curva de Tensión Escalonada

Dificultad y economía se dosifican por bloques/eventos apilados en la partida actual, nunca por capítulo — cualquier Historia arranca igual de legible.

ARQ-116

Moneda Única

Retiro del sistema dual de divisas a favor de una sola moneda (Neon Coins) — skins, Historias y revivals se compran y ganan con el mismo saldo.

ARQ-117

AdManager productivo

Banner + Interstitial + Rewarded sobre un único plugin nativo, con estrategia antifrustración basada en partidas perdidas acumuladas.

Protocolo de razonamiento obligatorio (AI Thinking Protocol)

Antes de escribir código, todo agente debe validar mentalmente este checklist — documentado íntegro en CLAUDE.md:

1. Duplicación

¿Esta lógica ya existe en Engine.js o Assets.js?

2. GameState

¿Qué estructura de datos se ve alterada por esta función?

3. Memoria

¿Se instancian objetos que activen el GC en móviles?

4. Render puro

¿Afecta al renderizado puro del Canvas?

5. KISS

¿Es la solución más simple que cumple el requisito?
